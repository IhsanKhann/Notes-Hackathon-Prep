01

INFRASTRUCTURE AS CODE — GCLOUD CLI

Enabling APIs, authenticating, and configuring your project environment from the terminal

↳ Why This Step Exists

GCP services are disabled by default to prevent accidental usage and billing. Enabling an API is like flipping a circuit breaker — nothing can talk to Cloud Run, Artifact Registry, or Secret Manager until you explicitly activate the service. This is not optional; without it every subsequent command will return a SERVICE_DISABLED error. The CLI approach is faster than the Console and is version-controllable — you can script your entire environment setup as an onboarding script for teammates.

STEP A — AUTHENTICATE AND SET PROJECT

POWERSHELL / CMD — run in order
# 1. Open Google Cloud SDK Shell (installed with the SDK)
#    OR use PowerShell if gcloud is in your PATH

# 2. Authenticate — opens a browser window
gcloud auth login

# 3. Set application-default credentials (needed for local SDK calls)
gcloud auth application-default login

# 4. List your projects to find your PROJECT_ID
gcloud projects list

# 5. Set the active project (replace with YOUR actual project ID)
gcloud config set project YOUR_PROJECT_ID

# 6. Set default region — us-central1 has the most free tier quota
gcloud config set run/region us-central1

# 7. Confirm your config
gcloud config list
⚠️
Windows: 'gcloud' is not recognized

The SDK installer adds gcloud to PATH only if you checked that option. Fix: Open "Google Cloud SDK Shell" from Start Menu instead of regular PowerShell. Alternatively, add C:\Users\YOUR_USER\AppData\Local\Google\Cloud SDK\google-cloud-sdk\bin to your system PATH manually via System Properties → Environment Variables.

⚠️
Windows: Browser doesn't open for auth login

Run gcloud auth login --no-launch-browser. It prints a URL — copy it into any browser manually, authenticate, then paste the authorization code back into the terminal.

STEP B — ENABLE THE THREE REQUIRED APIs

↳ What Each API Does

run.googleapis.com — the Cloud Run service itself. Without it, gcloud run deploy does nothing.
artifactregistry.googleapis.com — stores your Docker image. Cloud Run pulls from here, not from Docker Hub. It's private, faster within GCP, and free for 500MB.
secretmanager.googleapis.com — the vault for your API keys and MongoDB URIs. Never store these in environment variables passed directly — use Secret Manager instead.

POWERSHELL — enable all three in one command
gcloud services enable \
  run.googleapis.com \
  artifactregistry.googleapis.com \
  secretmanager.googleapis.com

# On Windows CMD (no backslash continuation) — use this instead:
gcloud services enable run.googleapis.com artifactregistry.googleapis.com secretmanager.googleapis.com
VERIFY — confirm all three are active
gcloud services list --enabled --filter="name:(run OR artifactregistry OR secretmanager)"

# Expected output:
# NAME                              TITLE
# artifactregistry.googleapis.com  Artifact Registry API
# run.googleapis.com               Cloud Run Admin API
# secretmanager.googleapis.com     Secret Manager API

STEP C — CREATE AN ARTIFACT REGISTRY REPOSITORY

POWERSHELL — create the Docker image repository
gcloud artifacts repositories create project-test-repo \
  --repository-format=docker \
  --location=us-central1 \
  --description="Docker images for Project Test FastAPI"

# Verify it was created:
gcloud artifacts repositories list --location=us-central1

# Configure Docker to authenticate to GCP (run once per machine):
gcloud auth configure-docker us-central1-docker.pkg.dev

us-central1-docker.pkg.dev/YOUR_PROJECT_ID/project-test-repo/project-test:latest

SERVERLESS DEPLOYMENT — CLOUD RUN

Dockerfile → Artifact Registry → Cloud Run. The full pipeline explained and implemented.

↳ Why Cloud Run (Not App Engine, Not GKE)

Cloud Run is request-driven and scales to zero — you pay nothing when no requests come in. For a hackathon or test project, this means your free tier lasts far longer. App Engine always has one instance running (minimum billing). GKE requires running nodes 24/7. Cloud Run is the right choice: it handles your FastAPI container with zero infrastructure management, auto-scales on demand, and stays within free tier for low-traffic projects (first 2 million requests/month are free).

The relationship: Docker packages your app + Python runtime into a portable image. Artifact Registry is GCP's private Docker Hub — it stores that image inside Google's network (faster pulls, no rate limits). Cloud Run pulls that image and runs it as a serverless container, giving you an HTTPS URL automatically.

STEP B — WRITE THE DOCKERFILE

↳ Why Each Line Exists

The Dockerfile is a recipe. Cloud Run doesn't know what Python is — it only knows how to run containers. The Dockerfile teaches it: "start from a Python image, install these packages, copy this code, then start uvicorn on port 8080." Port 8080 is mandatory for Cloud Run (not 8000, not 3000).

Dockerfile — place in Project_test/ root (same level as main.py)
# ── Base image: slim = smaller image = faster cold start ──
FROM python:3.11-slim

# ── Working directory inside the container ──
WORKDIR /app

# ── Install deps BEFORE copying code (Docker layer caching) ──
# If only main.py changes, this layer is reused — faster builds
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# ── Copy your application code ──
COPY main.py .

# ── Cloud Run REQUIRES port 8080 ──
# The PORT env var is injected by Cloud Run automatically
ENV PORT 8080
EXPOSE 8080

# ── Start uvicorn — listen on 0.0.0.0 so Cloud Run can reach it ──
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8080"]

STEP C — FASTAPI APP WITH ENVIRONMENT VARIABLE HANDLING

↳ Why Environment Variables for MERN-style Architecture

In a MERN stack the Node.js backend uses process.env.MONGO_URI. The Python/FastAPI equivalent is os.getenv("MONGO_URI"). The pattern is identical: never hardcode connection strings. Locally, they come from your .env file. In Cloud Run, they come from Secret Manager or environment variable injection at deploy time. The python-dotenv package bridges both worlds with zero code change.

main.py — production-ready FastAPI with env var handling
import os
from fastapi import FastAPI
from dotenv import load_dotenv

# ── Loads .env locally; has NO effect when deployed to Cloud Run ──
# (Cloud Run injects vars directly — no .env file exists there)
load_dotenv()

app = FastAPI(
    title="Project Test API",
    version="1.0.0"
)

# ── Read environment variables ──
# os.getenv(key, default) — second arg is fallback if var not set
MONGO_URI     = os.getenv("MONGO_URI",     "mongodb://localhost:27017")
APP_ENV       = os.getenv("APP_ENV",       "development")
NEWS_API_KEY  = os.getenv("NEWS_API_KEY",  "")

# ── Health check — Cloud Run calls this to verify your app started ──
@app.get("/")
async def health_check():
    return {
        "status": "ok",
        "environment": APP_ENV,
        # Never return the actual secret value — just confirm it's set
        "mongo_configured": bool(MONGO_URI),
    }

@app.get("/hello")
async def hello_world():
    return {"message": "Hello from Cloud Run!", "env": APP_ENV}

# ── LOCAL ONLY — uvicorn from IDE or python main.py ──
if __name__ == "__main__":
    import uvicorn
    uvicorn.run("main:app", host="0.0.0.0",
                port=8000, reload=True)

ENTERPRISE SECURITY — SECRET MANAGER

Storing secrets, retrieving them in Python, and the IAM roles that prevent 403 errors

↳ Why Not Just Use Environment Variables?

You can pass env vars directly to Cloud Run at deploy time. The problem: they appear in plaintext in your shell history, in deployment logs, and in CI/CD pipeline output. Secret Manager encrypts secrets at rest with Google-managed keys, provides full audit logs of every access, allows secret rotation without redeployment, and lets you control which service accounts can read which secrets. For anything involving database credentials or API keys, Secret Manager is the professional standard — not optional.

STEP A — STORE SECRETS (THE "ekStore" PROCESS)

POWERSHELL — create and store secrets via CLI
# ── Method 1: Create from a string value (PowerShell) ──
$MONGO_URI = "mongodb+srv://user:password@cluster.mongodb.net/mydb"
$MONGO_URI | gcloud secrets create MONGO_URI --data-file=-

# ── Method 2: Create from a file ──
# First write your secret to a temp file:
"mongodb+srv://user:password@cluster.mongodb.net/mydb" | Out-File -NoNewline mongo_uri.txt
gcloud secrets create MONGO_URI --data-file=mongo_uri.txt
Remove-Item mongo_uri.txt   # delete immediately after

# ── Add more secrets for your hackathon stack ──
"your_news_api_key_here"       | gcloud secrets create NEWS_API_KEY     --data-file=-
"your_anthropic_key_here"      | gcloud secrets create ANTHROPIC_API_KEY --data-file=-
"your_openweather_key_here"    | gcloud secrets create OPENWEATHER_KEY   --data-file=-

# ── Verify secrets exist ──
gcloud secrets list

# ── View a secret's value (for debugging only) ──
gcloud secrets versions access latest --secret="MONGO_URI"

# ── Add a new version (rotate the secret) ──
"new_mongo_uri_value" | gcloud secrets versions add MONGO_URI --data-file=-

STEP B — IAM PERMISSIONS (PREVENTING 403 FORBIDDEN)

↳ Why You Get 403 Without This Step

Cloud Run runs your code under a Service Account identity. By default, this is the Compute Engine default service account. That account has no permission to read Secret Manager secrets. When your Python code calls the Secret Manager SDK, GCP checks: "does this service account have secretmanager.secretAccessor?" If not — 403 Forbidden. You must explicitly grant the role.

POWERSHELL — create a dedicated service account and grant roles
# ── Best practice: create a DEDICATED service account for your app ──
# (Don't use the default compute service account — too many permissions)

$PROJECT_ID = "your-project-id"
$SA_NAME    = "project-test-sa"
$SA_EMAIL   = "$SA_NAME@$PROJECT_ID.iam.gserviceaccount.com"

# 1. Create the service account
gcloud iam service-accounts create $SA_NAME `
  --display-name="Project Test Service Account"

# 2. Grant Secret Manager accessor role — allows reading secret VALUES
gcloud projects add-iam-policy-binding $PROJECT_ID `
  --member="serviceAccount:$SA_EMAIL" `
  --role="roles/secretmanager.secretAccessor"

# 3. Redeploy Cloud Run using this service account
gcloud run deploy project-test `
  --region us-central1 `
  --service-account $SA_EMAIL `
  --image "us-central1-docker.pkg.dev/$PROJECT_ID/project-test-repo/project-test:latest"
IAM RoleWhat It AllowsWhen You Need It
roles/secretmanager.secretAccessorRead secret values (access versions)Always — your app needs this
roles/secretmanager.viewerList secrets, see metadata (not values)Admin/monitoring accounts only
roles/secretmanager.adminCreate, delete, manage all secretsNever assign to app service account
roles/run.invokerCall your Cloud Run service via HTTPCI/CD service account for testing

secrets.py — dedicated module for secret retrieval (modular architecture)
import os
from google.cloud import secretmanager
from dotenv import load_dotenv

load_dotenv()

# ── Cache client — creating it is expensive, reuse it ──
_sm_client = None

def _get_client() -> secretmanager.SecretManagerServiceClient:
    global _sm_client
    if _sm_client is None:
        _sm_client = secretmanager.SecretManagerServiceClient()
    return _sm_client


def get_secret(secret_id: str, version: str = "latest") -> str:
    """
    Retrieve a secret value from Google Cloud Secret Manager.

    Locally: falls back to .env file via os.getenv first.
    In Cloud Run: uses the Secret Manager API (service account auth).

    Args:
        secret_id: The secret name as created in Secret Manager (e.g. "MONGO_URI")
        version:   Usually "latest". Can be a version number like "3".

    Returns:
        The secret value as a string.

    Raises:
        google.api_core.exceptions.NotFound: Secret doesn't exist.
        google.api_core.exceptions.PermissionDenied: 403 — missing IAM role.
    """
    # ── Local development shortcut ──
    # If running locally and .env has the var, use it directly.
    # Avoids needing internet access for local dev.
    local_val = os.getenv(secret_id)
    if local_val:
        return local_val

    # ── Cloud Secret Manager path ──
    project_id = os.getenv("GOOGLE_CLOUD_PROJECT") or os.getenv("GCP_PROJECT")
    if not project_id:
        raise ValueError(
            "GOOGLE_CLOUD_PROJECT env var not set. "
            "Add it to your .env or Cloud Run config."
        )

    client = _get_client()
    name   = f"projects/{project_id}/secrets/{secret_id}/versions/{version}"

    response = client.access_secret_version(request={"name": name})
    return response.payload.data.decode("UTF-8")


# ── Typed accessors — cleaner than raw string calls everywhere ──
def get_mongo_uri() -> str:
    return get_secret("MONGO_URI")

def get_news_api_key() -> str:
    return get_secret("NEWS_API_KEY")

def get_anthropic_key() -> str:
    return get_secret("ANTHROPIC_API_KEY")
main.py — using secrets.py in your FastAPI app
import os
from fastapi import FastAPI
from dotenv import load_dotenv
from secrets import get_mongo_uri, get_news_api_key

load_dotenv()

# ── Fetch secrets ONCE at startup, not on every request ──
# This is the correct pattern — avoids API call on each request

@app.get("/")
async def health():
    return {
        "status": "ok",
        "mongo_connected": bool(MONGO_URI),
        "news_api_ready":   bool(NEWS_API_KEY),
    }

PRODUCTION CI/CD — GITHUB ACTIONS

Workload Identity Federation: zero long-lived keys, maximum security, auto-deploy on push

↳ Why Workload Identity Federation Instead of Service Account Keys

The naive approach: create a service account, download a JSON key file, paste it as a GitHub Secret. The problem: that key file is valid forever, can be leaked, and if compromised gives an attacker permanent access. Workload Identity Federation eliminates the key file entirely. Instead, GitHub's OIDC token (a short-lived JWT proving "this is a GitHub Actions run from repo X") is exchanged for a temporary GCP access token that expires in 1 hour. No static credentials. No key rotation headache. This is what Google's own engineers use.

STEP A — SET UP WORKLOAD IDENTITY FEDERATION

Run these commands ONCE. They create the trust relationship between your GitHub repo and GCP.

POWERSHELL — WIF setup (run once, then keep the output values)
# ── Set your variables ──
$PROJECT_ID    = "your-project-id"
$PROJECT_NUM   = "123456789012"           # get from: gcloud projects describe $PROJECT_ID
$GITHUB_ORG    = "your-github-username"   # or org name
$GITHUB_REPO   = "your-repo-name"
$POOL_ID       = "github-actions-pool"
$PROVIDER_ID   = "github-actions-provider"
$SA_NAME       = "project-test-sa"
$SA_EMAIL      = "$SA_NAME@$PROJECT_ID.iam.gserviceaccount.com"

# ── 1. Create Workload Identity Pool ──
gcloud iam workload-identity-pools create $POOL_ID `
  --project=$PROJECT_ID `
  --location=global `
  --display-name="GitHub Actions Pool"

# ── 2. Add GitHub as an OIDC provider in the pool ──
gcloud iam workload-identity-pools providers create-oidc $PROVIDER_ID `
  --project=$PROJECT_ID `
  --location=global `
  --workload-identity-pool=$POOL_ID `
  --display-name="GitHub Actions Provider" `
  --attribute-mapping="google.subject=assertion.sub,attribute.actor=assertion.actor,attribute.repository=assertion.repository" `
  --issuer-uri="https://token.actions.githubusercontent.com"

# ── 3. Allow GitHub Actions from YOUR repo to impersonate the service account ──
# This is the key security control — only THIS specific repo gets access
gcloud iam service-accounts add-iam-policy-binding $SA_EMAIL `
  --project=$PROJECT_ID `
  --role="roles/iam.workloadIdentityUser" `
  --member="principalSet://iam.googleapis.com/projects/$PROJECT_NUM/locations/global/workloadIdentityPools/$POOL_ID/attribute.repository/$GITHUB_ORG/$GITHUB_REPO"

# ── 4. Grant the service account permission to deploy to Cloud Run ──
gcloud projects add-iam-policy-binding $PROJECT_ID `
  --member="serviceAccount:$SA_EMAIL" `
  --role="roles/run.admin"

gcloud projects add-iam-policy-binding $PROJECT_ID `
  --member="serviceAccount:$SA_EMAIL" `
  --role="roles/artifactregistry.writer"

gcloud projects add-iam-policy-binding $PROJECT_ID `
  --member="serviceAccount:$SA_EMAIL" `
  --role="roles/iam.serviceAccountUser"

# ── 5. Get the Workload Identity Provider resource name ──
# SAVE THIS OUTPUT — you need it for the GitHub Actions workflow
gcloud iam workload-identity-pools providers describe $PROVIDER_ID `
  --project=$PROJECT_ID `
  --location=global `
  --workload-identity-pool=$POOL_ID `
  --format="value(name)"

STEP C — THE COMPLETE deploy.yml

Create this file at Project_test/.github/workflows/deploy.yml

.github/workflows/deploy.yml — full production pipeline
name: Build and Deploy to Cloud Run

# ── Trigger: only on pushes to main ──
on:
  push:
    branches:
      - main

# ── Permissions required for Workload Identity Federation ──
# id-token: write — allows GitHub to issue OIDC token
# contents: read — allows checkout of your repository
permissions:
  contents: read
  id-token: write

env:
  REGION:      us-central1
  REPO:        project-test-repo
  SERVICE:     project-test
  IMAGE:       us-central1-docker.pkg.dev/${{ secrets.GCP_PROJECT_ID }}/project-test-repo/project-test

jobs:
  deploy:
    name: Build, Push, Deploy
    runs-on: ubuntu-latest

    steps:

      # ── 1. Checkout your code ──
      - name: Checkout repository
        uses: actions/checkout@v4

      # ── 2. Authenticate to GCP via Workload Identity Federation ──
      # No JSON key. GitHub's OIDC token is exchanged for a GCP token.
      - name: Authenticate to Google Cloud
        id: auth
        uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: ${{ secrets.WIF_PROVIDER }}
          service_account:            ${{ secrets.WIF_SERVICE_ACCOUNT }}

      # ── 3. Install gcloud CLI on the runner ──
      - name: Set up Cloud SDK
        uses: google-github-actions/setup-gcloud@v2

      # ── 4. Configure Docker to push to Artifact Registry ──
      - name: Configure Docker for Artifact Registry
        run: gcloud auth configure-docker ${{ env.REGION }}-docker.pkg.dev --quiet

      # ── 5. Build the Docker image ──
      # Uses GitHub SHA as tag for exact version tracking + "latest"
      - name: Build Docker image
        run: |
          docker build \
            -t ${{ env.IMAGE }}:${{ github.sha }} \
            -t ${{ env.IMAGE }}:latest \
            .

      # ── 6. Push both tags to Artifact Registry ──
      - name: Push Docker image
        run: |
          docker push ${{ env.IMAGE }}:${{ github.sha }}
          docker push ${{ env.IMAGE }}:latest

      # ── 7. Deploy the new image to Cloud Run ──
      # Uses the SHA-tagged image — pinned, not floating "latest"
      - name: Deploy to Cloud Run
        run: |
          gcloud run deploy ${{ env.SERVICE }} \
            --image ${{ env.IMAGE }}:${{ github.sha }} \
            --region ${{ env.REGION }} \
            --platform managed \
            --service-account ${{ secrets.WIF_SERVICE_ACCOUNT }} \
            --set-env-vars GOOGLE_CLOUD_PROJECT=${{ secrets.GCP_PROJECT_ID }},APP_ENV=production \
            --memory 512Mi \
            --cpu 1 \
            --min-instances 0 \
            --max-instances 3 \
            --allow-unauthenticated \
            --quiet

      # ── 8. Print the live URL ──
      - name: Get Service URL
        run: |
          gcloud run services describe ${{ env.SERVICE }} \
            --region ${{ env.REGION }} \
            --format="value(status.url)"
1
actions/checkout@v4

Clones your repository into the runner VM. Without this, subsequent steps have no files to work with.

2
google-github-actions/auth@v2 (the WIF handshake)

GitHub creates a signed OIDC token containing your repo name. It sends this to GCP's Security Token Service. GCP verifies the token matches the trusted pool + provider config you set up in Step A. If it matches, GCP issues a short-lived access token. No JSON key ever touched disk.

3–4
setup-gcloud + configure-docker

Installs the gcloud CLI on the Ubuntu runner and configures Docker to use GCP credentials when pushing to us-central1-docker.pkg.dev.

5–6
Build + Push (dual-tagged)

Builds and pushes two tags: :latest for human reference, :github.sha (e.g. :a3f8c1d) for pinned deployment. Using the SHA tag means Cloud Run always runs the exact commit that triggered the deploy — not a floating "latest" that could be overwritten mid-deploy.

Testing the pipeline

Make a trivial change to main.py (update the version string), commit and push to main. Go to your GitHub repo → Actions tab. You'll see the workflow running. Green checkmarks on all 8 steps = successful deploy. The final step prints your live URL.

🚫
Common CI/CD Failure: "Permission denied on workload identity"

This means the principalSet member string in Step A doesn't exactly match your repo. Verify: $GITHUB_ORG/$GITHUB_REPO must match the exact case-sensitive GitHub path (e.g. JohnDoe/my-project not johndoe/my-project). Also confirm id-token: write appears in your workflow permissions block — it's required for the OIDC token to be issued.

⊕ End-to-End Workflow Summary — your daily development loop

Local dev: Edit main.py → run uvicorn main:app --reload → secrets come from .env.

Test it locally in Docker: docker build -t project-test . && docker run -p 8080:8080 --env-file .env project-test → visit localhost:8080.

Deploy to production: git add . && git commit -m "feat: your change" && git push origin main → GitHub Actions does the rest automatically in ~3 minutes. No manual gcloud commands.
